## 2. Project Context And Final Scope

### 2.1 Problem Context

NeighbourLink addresses a common suburban mobility problem: many riders need reliable point-to-point transport for work or one-off trips, while drivers often have spare seat capacity. The system focuses on matching, confirmation, and trust visibility rather than full commercial ride-hailing complexity.

The project intentionally avoids out-of-scope commercial features (payment processing, chat, route optimization engines, and complex role/permission matrices) to keep the assignment bounded and technically coherent.

### 2.2 Scope Boundary (Final)

In the final version, the primary flows are still the three approved use cases:

1. Search Available Ride Offers
2. Request to Join a Ride Offer
3. Post a One-Off Ride Request and Accept Driver Offer

Important supporting capability added in the final implementation:

- Trust-first acceptance context (profile + rating summary before acceptance decisions)
- Admin governance panel for all system records (fixed admin account only)
- Driver document upload and admin verification workflow
- Notifications and trip history views
- Map-assisted location selection and location lookup

### 2.3 Final Technology Footprint

Backend:
- Java 17, Spring Boot
- Spring Web, Spring Data JPA, Bean Validation
- SQLite with JPA schema update mode

Frontend:
- Vite build tool
- Native JavaScript single-page shell (non-React runtime)
- Leaflet for map interaction

Operational model:
- Local deployment
- One-command startup script for backend + frontend orchestration

## 4. UML-Driven Analysis Across Multiple Dimensions

The following sections explain how UML artifacts are interpreted not as static diagrams, but as design commitments that shape code structure and behavior.

## 4.1 Use Case Diagram Interpretation

Given actors:
- Rider
- Driver

Main use cases:
- UC1 Search Available Ride Offers
- UC2 Request to Join Ride Offer
- UC3 Post One-Off Ride Request

Supporting use cases:
- View Driver Profile and Rating
- Review/Accept/Reject Join Request
- View Posted Ride Requests
- Submit Ride Request Offer
- Accept Driver Offer
### 4.1.1 Semantic Role Of <<include>> And <<extend>>

In this project, include/extend relationships are operationally meaningful:

- UC2 includes driver review flow, because rider submission is incomplete until a driver decision is produced.
- UC4 extends acceptance contexts, because trust review is conditionally invoked before a commitment action.
- UC10 extends UC9, because rider acceptance depends on existing driver offers.

This interpretation prevents a common modeling mistake: treating every use case as isolated screens. Instead, use cases become transaction pipelines with preconditions and postconditions.

### 4.1.2 Actor Responsibility Boundaries

Rider responsibilities:
- Search and submit intent
- Evaluate trust data
- Accept one-off offers when presented

Driver responsibilities:
- Review inbound rider requests
- Accept/reject join requests
- Respond to one-off rider demand

System responsibilities:
- Validate state and seat constraints
- Persist decisions as auditable records
- Create RideMatch only after explicit acceptance
- Trigger notifications and history updates

This boundary is mirrored in backend package responsibilities (controllers as request boundary, services as state transition logic, repositories as persistence).

## 4.2 Activity Diagram Interpretation (Scenario 1)

The activity diagram emphasizes three critical checkpoints:

1. Match discovery and selection
2. Trust-informed rider request submission
3. Driver decision leading to either rejection or confirmed match

### 4.2.1 Process Control Semantics

Key decision nodes:
- Request this ride? (rider intent decision)
- Ride open and seats available? (system guard)
- Accept request? (driver authority)

Each node maps to a different type of control:

- User intent gate
- System invariant gate
- Counterparty gate

This tri-gate structure is important. It avoids false assumptions that a submitted request equals a confirmed trip.

### 4.2.2 State Transition Guarantees

A robust implementation enforces these transitions:

- JoinRequest: PENDING -> ACCEPTED or REJECTED
- RideOffer: OPEN remains OPEN until seats/conditions force closure
- RideMatch: created only on ACCEPTED path

The diagram’s stop points for rejection and unavailable seats are implemented as explicit response outcomes, not hidden exceptions.

## 4.3 Domain/Class Diagram Interpretation

Your final class diagram moves beyond flat associations and introduces richer relationship semantics:

- Generalization: Rider and Driver inherit from User
- Composition-like ownership: User owns Credential and optionally Profile
- Aggregation-like groupings: RideOffer receives JoinRequests; RideRequest receives RideRequestOffers
- Explicit provenance links: RideMatch references accepted JoinRequest or accepted RideRequestOffer

### 4.3.1 Why Relationship Type Matters

Generalization supports role-specific attributes without duplicating common identity fields.

Ownership/composition semantics make lifecycle intent clearer:
- Credentials should not exist independently of a user.
- Profile and trust context are logically tied to user identity.

Aggregation signals collection relationships where child records are independent transactions, but still belong to a parent process context.

### 4.3.2 Business Constraints Encoded In Model

Important constraints from the model are now implementation-relevant:

- requestedSeats <= availableSeats
- RideMatch XOR provenance (exactly one acceptance source path)
- Concurrency-aware offer decisioning
- Ratings include rater linkage (`raterUser`)

These constraints transform UML from communication artifacts into enforceable design contracts.

## 4.4 Derived UML Perspectives For Presentation Depth

Even if not explicitly drawn in final submission, two additional UML perspectives can strengthen your narrative:

### 4.4.1 Sequence Perspective

Sequence for UC2:
- Rider UI -> RideOfferQuery API -> Offer detail + trust summary
- Rider UI -> JoinRequest API -> JoinRequestService -> persistence
- Driver UI -> Decision API -> service with lock/validation
- Service -> RideMatch creation + seat update + notification events

Sequence for UC3:
- Rider UI -> OneOffRequest API
- Driver UI -> submit offer API
- Rider UI -> accept offer API
- Service -> close request + create match + notify

### 4.4.2 State Machine Perspective

RideOffer state machine:
- OPEN -> CLOSED (capacity exhausted or manually/admin updated)

JoinRequest state machine:
- PENDING -> ACCEPTED | REJECTED

RideRequest state machine:
- OPEN -> MATCHED | CLOSED

RideRequestOffer state machine:
- PENDING -> ACCEPTED | REJECTED

RideMatch state machine:
- CONFIRMED -> COMPLETED | CANCELLED

These state machines help explain why separate history tabs exist in My Trips: they expose business process states to the user.

## 6. Architecture, Design Trade-Offs, And Rationale

## 6.1 Why Monolith + Layered Architecture Was Appropriate

For assignment constraints and local deployment, a layered monolith was the right architecture:

- Faster iteration and easier end-to-end debugging
- Lower operational complexity than microservices
- Clear educational mapping from UML to code packages
- Sufficient scalability for prototype and scenario-driven validation

The architecture keeps a clean separation:
- Controller layer: HTTP contract + validation boundary
- Service layer: domain logic and transition orchestration
- Repository layer: persistence queries and locking semantics
- Entity layer: domain model and relational mapping

## 6.2 Trade-Off: Simplicity vs Real-World Completeness

What was intentionally not implemented:
- Payment settlement
- In-app messaging/chat
- Live route ETA optimization
- Role-permission matrix beyond fixed admin and user roles

Why this is correct for scope:
- These features would dilute core use case quality.
- Trust, matching, and confirmation correctness are more central to assignment objectives.

## 6.3 Frontend Trade-Off: Native JS SPA vs Component Framework

The migration to native JS was aligned to project requirement preferences.

Advantages:
- Zero framework runtime dependency
- Direct control over rendering and state flow
- Better transparency for assignment review

Costs:
- More manual state wiring
- Less built-in component reuse
- Higher discipline needed for maintainability

Mitigation:
- Centralized render/router model
- Dedicated API adapter module
- Naming conventions and route-scoped render functions

## 7. End-To-End Business Logic Validation

## 7.1 UC1 Validation Logic

Input dimensions:
- origin, destination, date, passenger count
- optional time and flexibility window

Matching behavior:
- Search open offers
- Apply seat sufficiency checks
- Apply suburb-level fallback and synonym matching at frontend orchestration level

Presentation behavior:
- Show summary and no-result fallback actions
- Route to details for trust-informed request decision

## 7.2 UC2 Validation Logic

Critical checks on request submission and decision:

- Ride offer exists and is open
- Requested seats are valid positive integer
- Requested seats <= current available seats
- Decision operation verifies current status and availability under concurrency controls
- On acceptance:
  - JoinRequest becomes ACCEPTED
  - RideMatch created with accepted JoinRequest provenance
  - Seat count updated
  - Notifications/history become visible
- On rejection:
  - JoinRequest becomes REJECTED
  - Rider sees outcome in history and notifications

## 7.3 UC3 Validation Logic

Process phases:

- Rider posts one-off request with route/time/passenger constraints
- Driver submits offer with proposed seats and meeting point
- Rider reviews offers with trust data and accepts one

Critical consistency behavior:

- Only one accepted offer should become match source
- RideMatch created with accepted RideRequestOffer provenance
- Request status transitions and competing pending offers handled appropriately
- Notifications and My Trips updated to reflect final state

## 7.4 Trust-Before-Acceptance Compliance

The model requires trust context before acceptance actions.

Implemented trust artifacts:
- Profile summary
- Rating average and count
- Rating comments and metadata access path

Trust exposure points:
- Before submitting join request on ride offer details
- Before rider accepts a one-off driver offer

This is a high-value alignment point for report and demo because it directly ties supporting process requirements to UI and API behavior.

## 7.5 Concurrency And Integrity Risk Mitigation

Potential risk:
- Two near-simultaneous acceptance actions consuming the same seats or matching source.

Mitigation implemented:
- Versioning and lock-based offer retrieval on decision paths
- Database uniqueness and check constraints for match provenance
- Status transition guards in service logic

Evidence framing for report:
- Business rule correctness is enforced at multiple levels: application service, repository lock semantics, and database constraints.

### Part F: Trust and rating closure

1. In rider or driver My Trips history, show rating submission context.

### 12.1 Introduction

NeighbourLink is a web-based prototype that supports community ride coordination through structured matching and confirmation flows. The system is intentionally scoped to three approved use cases and is designed around two complementary process goals: core matching and trust-aware decision support. The project is implemented with a Spring Boot backend and a native JavaScript frontend, with SQLite persistence.

The technical and academic objective was not simply to build pages and endpoints, but to establish clear UML-to-code alignment. This means each major design construct in use case, activity, and class diagrams should be observable in runtime behavior, state transitions, and persistence constraints.

### 12.2 Requirement And Scope Definition

The final solution adopts constrained scope as a quality strategy. Payments, direct messaging, and advanced route optimization were excluded deliberately because they are orthogonal to assignment intent and would increase complexity without improving traceability. Instead, effort focused on ensuring correctness of acceptance paths, visibility of trust data before commitment, and integrity of match provenance.

This scoped strategy provided two benefits. First, it made the codebase easier to reason about for markers and reviewers. Second, it enabled deeper implementation quality on business-critical transitions, such as preventing seat over-allocation and ensuring one match corresponds to one accepted source pathway.

### 12.3 UML-Driven Design Method

The use case diagram defines actor responsibilities and interaction contracts. Rider initiates search and requests, driver reviews and responds, and system validates transitions and records agreements. The include and extend semantics were treated as behavior contracts rather than visual decoration, especially in trust exposure and decision dependencies.

The activity diagram for Scenario 1 was treated as a strict process template. The implementation maintains explicit branch outcomes for unavailable rides and rejected requests, and only creates a match on acceptance branches. This removes ambiguity between intention and agreement.

The class diagram was expanded to represent richer structural semantics. Beyond basic associations, the design communicates inheritance for user roles, ownership of credentials and profile, collection relationships for request aggregation, and explicit references from RideMatch to accepted sources. These structural decisions drive both API and data integrity design.

### 12.4 Implementation Architecture

Backend architecture follows layered design:
- Controllers define API boundaries and request contracts.
- Services implement state transitions and business guards.
- Repositories provide persistence and query semantics, including lock pathways.
- Entities represent domain state and relational constraints.

Frontend architecture uses native JavaScript route rendering with centralized state. This supports assignment requirements while preserving flow traceability. Each main route corresponds to a process state in the UML narrative.

### 12.5 Data Integrity, Concurrency, And Provenance

Integrity controls are implemented at multiple levels. Service-level validation checks request correctness and state compatibility. Repository-level lock and query semantics protect critical decision paths. Database-level constraints prevent ambiguous or duplicate match source links.

The RideMatch model enforces source exclusivity through check constraints and unique constraints. This is important in Stage 2 because it directly addresses a common review concern: "Can you prove each match came from one explicit acceptance path?"

Concurrency risk in seat decisions is mitigated with version and locked decision retrieval pathways. While this does not transform the prototype into a distributed scheduling platform, it is a proportionate and academically strong mitigation for assignment scale.

### 12.6 Trust As A First-Class Process Component

Trust data is intentionally integrated before acceptance actions. In rider decision surfaces, profile and rating summaries are shown before request or acceptance submission. This design aligns with the supporting process objective from UML notes and elevates trust from passive metadata to active decision support.

### 12.7 Prototype Outcome Summary

The final prototype demonstrates all three use cases in runnable browser workflows with status visibility and confirmation evidence. Rider and driver histories expose transition outcomes, notifications communicate agreement events, and admin views provide governance-level observability.

From an assessment perspective, this means the solution is not only functionally complete but also explainable. Markers can inspect route behavior, API responses, and persistence effects in a consistent chain.

### 12.8 Limitations And Practical Boundaries

The project still has known boundaries:
- No payment or financial settlement logic
- No built-in chat negotiation channel
- No production-grade identity federation
- No distributed scalability architecture

These limitations are acceptable within assignment scope, but they should be acknowledged transparently in final reporting.

### 12.9 Future Work Within Controlled Scope

- richer ranking strategy for search relevance

Future work should preserve the same design discipline: every extension should be traceable to updated model artifacts and enforced invariants.

## 26. System Structure Deep Dive: From Conceptual Architecture To Runtime Topology

This section expands the structural explanation beyond high-level architecture labels. The objective is to show how NeighbourLink is physically and logically assembled at runtime, why each boundary exists, and how the final composition supports the assignment’s core use cases without introducing unnecessary complexity.

### 26.1 Structural Philosophy

NeighbourLink was implemented with a “minimum sufficient structure” mindset. This means:

- avoid architecture that exceeds assignment constraints
- maximize explainability and traceability
- make state transitions inspectable
- keep deployment and demonstration deterministic

The project structure intentionally favors layered clarity over platform abstraction. In practical terms, this allows reviewers to trace a user action in UI to endpoint, service, repository query, entity mutation, and persisted result with minimal cognitive overhead.

### 26.2 Runtime Topology

At runtime, the system is composed of two local processes:

1. Spring Boot backend (port 8080)
2. Vite frontend development server (port 5173)

The frontend behaves as a native JavaScript SPA shell. It requests backend REST resources directly. There is no extra BFF (backend-for-frontend), no API gateway, and no event broker in this prototype.

Why this topology is appropriate:
- one machine local deployment requirement
- assignment-scale reliability needs
- clear demonstration setup with low operational friction

### 26.3 Source Tree As Architectural Contract

The repository layout is not only for convenience; it encodes responsibilities.

Top-level responsibilities:
- `backend/`: all domain behavior and persistence
- `frontend/`: all user interaction and flow orchestration
- `docs/`: API and report-facing artifacts
- startup scripts: deterministic local bootstrapping

The backend package layout reflects an explicit vertical flow:
- `controller`: boundary contract
- `service`: business orchestration
- `repository`: persistence behavior
- `entity`: domain model and relational mapping
- `dto`: API-facing data transfer model
- `exception`: consistent error semantics
- `config`: crypto/cors and application boundaries

This layout is essential for UML traceability because each diagram concern maps to one or more concrete package responsibilities.

### 26.4 Boundary of Responsibility: Frontend vs Backend

A common failure mode in student projects is unclear ownership between frontend and backend validations. NeighbourLink uses the following boundary contract:

Frontend responsibilities:
- interaction flow sequencing
- user guidance and immediate form constraints
- rendering status/result contexts
- preserving stateful navigation context

Backend responsibilities:
- authoritative rule enforcement
- transition legality checks
- race condition mitigation
- persistence integrity

This contract means frontend can improve UX, but cannot redefine business truth. If frontend misses a check, backend still protects correctness.

### 26.5 Data Ownership Domains

The data model can be separated into four ownership domains:

Identity domain:
- User, Rider, Driver, Credential

Trust domain:
- Profile, Rating

Transaction domain:
- RideOffer, JoinRequest, RideRequest, RideRequestOffer, RideMatch

Operational domain:
- Notification, location references, driver documents

By organizing data this way, the system avoids conflating identity with transaction state. This is particularly important when explaining class diagram semantics in final report.

### 26.6 UI Structural Composition

Even without React components, the frontend has repeatable structural patterns:

- shared top-level layouts (`public`, `user`, `admin`)
- route-based render functions
- centralized state object with sub-state by feature area
- reusable interaction wiring helpers (navigation, map setup, location panels, paging)

This forms a pseudo-component architecture in plain JS. The design choice preserves requirement compliance (native stack) while retaining maintainability characteristics typically associated with component frameworks.

### 26.7 Structural Stability Through Scripts

The startup scripts (`start-neighbourlink.ps1` and `.cmd`) are significant structural assets. They enforce:

- environment check (`java`, `mvn`, `node`, `npm`)
- dependency preparation
- process startup order
- readiness verification
- log and PID trace paths

From an engineering process standpoint, these scripts reduce demonstration risk by converting setup from manual choreography into deterministic procedure.

### 26.8 Why No Microservices Here

It is important to justify why the architecture is not decomposed into microservices.

Reasons:
- assignment does not require distributed scaling
- state transitions require clarity more than network isolation
- introducing inter-service communication would reduce traceability
- local-only demo value is higher with monolithic coherence

In marking terms, this is a deliberate and mature trade-off, not a missing feature.

### 26.9 Observability Structure

Observability exists through three layers:

User-level observability:
- history tabs
- status chips
- notification feeds

Admin-level observability:
- cross-entity tables
- KPI counters
- bulk and inline update capabilities

Developer-level observability:
- startup logs
- health endpoint
- integration tests and build output

This three-layer observability model is a strong demonstration advantage because it supports both user-facing and assessor-facing validation.

### 26.10 Structural Dependency Summary

Core dependency path:
UI action -> route renderer -> API adapter -> controller -> service -> repository -> entity state -> database constraints -> DTO response -> UI refresh

This path is repeatedly demonstrable in UC1/UC2/UC3 and is the backbone of your UML-to-implementation narrative.

### 26.11 Structural Risks And Mitigations

Risk: monolithic `main.js` complexity growth.
Mitigation: route-specific function grouping, helper extraction, naming discipline.

Risk: frontend state coupling.
Mitigation: segmented state slices (`findFlow`, `postFlow`, `myTrips`, `driverHub`, `admin`) and local handler scoping.

Risk: stale UI assumptions on status.
Mitigation: backend remains source of truth; frontend re-fetches for terminal contexts.

Risk: data model drift from diagrams.
Mitigation: explicit mapping sections in documentation and admin data inspection path.

NeighbourLink’s structure is intentionally conservative but technically disciplined. It prioritizes demonstrable process correctness, traceable boundaries, and low-risk operations. This is exactly the profile expected from a strong assignment-stage system that claims UML-driven implementation quality.

## 29. Business Workflow Engineered As State-Oriented Transitions

This section explains implementation logic as workflow, not endpoint list.

### 29.1 UC2 Workflow Lifecycle

Stage 1: Rider intent submission
- JoinRequest created with PENDING status.
- No RideMatch created at this stage.

Stage 2: Driver decision
- Driver reviews pending request.
- System revalidates current offer state and seat feasibility.

Stage 3a: Acceptance branch
- JoinRequest marked ACCEPTED.
- RideMatch created with acceptedJoinRequest reference.
- Available seats decremented.
- Notifications generated.

Stage 3b: Rejection branch
- JoinRequest marked REJECTED.
- No RideMatch created.
- Rider notified of outcome.

This workflow exactly mirrors activity diagram semantics.

### 29.2 UC3 Workflow Lifecycle

Stage 1: Rider one-off demand creation
- RideRequest created with OPEN state.

Stage 2: Driver offer responses
- Multiple RideRequestOffer rows may accumulate.
- Each initially PENDING.

Stage 3: Rider acceptance of one offer
- Selected offer transitions to ACCEPTED.
- RideMatch created with acceptedRideRequestOffer reference.
- Request transitions accordingly (matched/closed pathway semantics).

Stage 4: Resolution of competing paths
- Non-selected pending offers must no longer remain misleadingly actionable.
- History views reflect final acceptance outcome.

### 29.3 Notification Workflow

Notifications are emitted on key state transitions:
- join accepted/rejected
- one-off offer acceptance outcomes
- match confirmations

This is a lightweight event-style mechanism without adding full event bus complexity.

My Trips is not a single list. It is a workflow observability console:

- Trip filter exposes current vs historical commitments.
- Join request history exposes rider-intent lifecycle.
- One-off request history exposes negotiation lifecycle.
- Driver offer history exposes response lifecycle.
- Notification tabs expose event acknowledgment lifecycle.

These sections help assessors verify whether transitions actually occurred as claimed.

### 29.5 Admin Workflow

Admin workflow is governance-centered:

- inspect records across entity families
- identify anomalies
- apply controlled updates
- verify effect through overview counts and table views

Because admin is fixed-account and non-registerable, governance behavior is controlled and predictable for demonstration.

### 29.6 Location Workflow

Location capture workflow combines:

- typed search query

- map click reverse lookup
- manual field correction

This multi-mode capture flow improves data quality for origin/destination metadata and supports more realistic demonstration behavior.

### 29.7 Rating Workflow

Rating submission occurs in historical trip context (post interaction). It binds:
- target profile
- rater identity
- score/comment/timestamp

This creates longitudinal trust data that feeds pre-acceptance trust surfaces.

A strong workflow implementation has three properties:

NeighbourLink’s Stage 2 improvements targeted all three.

## 35. Engineering Details: Concurrency, Transactions, And Integrity Design

This section focuses on lower-level reliability mechanics.

### 35.1 Concurrency Threat Model

Key risk cases:
- two drivers/actions attempting conflicting seat decisions
- stale client views submitting outdated assumptions
- duplicate acceptance attempts on already resolved records

### 35.2 Version-Based Protection

RideOffer versioning supports conflict awareness when mutable decision paths overlap in time.

While not a complete distributed lock strategy, this is meaningful protection in assignment-scale concurrent mutation contexts.

### 35.3 Lock-Based Decision Path

Repository-level pessimistic lock retrieval is used for decision-critical offer operations. This reduces window for contradictory seat decisions under overlapping operations.

### 35.4 Transition Guard Design

Service methods enforce transition legality:
- cannot accept already resolved request
- cannot allocate unavailable seats
- cannot create match from invalid source context

Guard clauses are explicit and produce meaningful errors.

### 35.5 Database Constraint Backstop

Persistence constraints provide final safety net:
- unique constraints prevent duplicated accepted-source reuse
- check constraints enforce match source exclusivity

This is crucial because application bugs alone should not silently corrupt agreement provenance.

### 35.6 Idempotency Considerations

Certain actions are not strictly idempotent by design (for example acceptance transitions). However, status guards and uniqueness constraints prevent repeated operations from producing duplicate terminal artifacts.

### 35.7 Consistency vs Availability Trade-Off At Scope Level

The system favors transition consistency over maximal write throughput in decision-critical paths. For assignment context, this is the right trade-off.

### 35.8 Error Surface Quality

When transition is rejected, user receives actionable error message rather than generic failure. This supports workflow recovery and improves demonstration robustness.

### 35.9 Integrity Validation In Demo

During demo, integrity can be shown by:
- attempting invalid repeat acceptance
- showing controlled rejection/guard output
- verifying no duplicate match artifacts appear

### 35.10 Reliability Conclusion

Reliability strategy is layered and proportionate: optimistic version awareness, pessimistic lock path for critical decisions, service guards, and DB constraints.

## 39. Backend Implementation Blueprint: Detailed Engineering Walkthrough

### 39.1 Request Handling Pipeline (HTTP To Persistence)

For each REST request, backend execution follows a consistent pipeline:

1. Controller endpoint receives request and maps to DTO.
2. Validation and basic argument checks are performed.
3. Service method executes business transition logic.
4. Repository queries retrieve/update entities under required consistency semantics.
5. Service assembles response DTO and returns via controller.
6. Global exception handler shapes error payload if failure occurs.

This consistency is not accidental. It reduces unexpected branch behavior and makes debugging straightforward during demos.

### 39.2 Join Request Creation Path (UC2 Entry)

The join creation path is a good example of transition discipline.

Input:
- rider id
- ride offer id
- requested seats

Execution concerns:
- rider existence and role validity
- offer existence and open status
- requested seats range
- duplicate pending semantics

Mutation result:
- new JoinRequest persisted with PENDING state
- no RideMatch created yet

Why this matters:
- keeps intent and agreement separate
- preserves driver authority in the process

### 39.3 Join Decision Path (UC2 Commitment)

Driver decision is transition-critical. The implementation guards this path carefully.

Execution sequence conceptually:

1. Load target JoinRequest and related offer context.
2. Validate request still pending.
3. Validate offer still open and seats available.
4. If reject:
   - set JoinRequest status REJECTED
   - create rider notification
   - return rejection response
5. If accept:
   - set JoinRequest status ACCEPTED
   - decrement available seats
   - create RideMatch with acceptedJoinRequest reference
   - update offer state if capacity reached
   - create notifications
   - return acceptance response

Engineering nuance:
- read-check-write is vulnerable without concurrency controls; lock/version support narrows this risk.

### 39.4 One-Off Request Creation Path (UC3 Entry)

This path captures rider demand with richer route metadata:
- text route fields
- address and suburb details
- coordinate fields
- trip date/time
- passenger and notes fields

Result:
- RideRequest persisted as OPEN
- visible to eligible driver workflow surfaces

### 39.5 Driver One-Off Offer Submission Path

Execution concerns include:
- target ride request exists and is open
- driver is valid and active
- proposed seats are valid
- meeting point quality constraints

Mutation:
- RideRequestOffer persisted with PENDING state
- rider-side review page can display trust and offer details

### 39.6 Rider Acceptance Of One-Off Offer (UC3 Commitment)

Commitment sequence:

1. Validate request and offer link.
2. Validate offer still pending and request still accept-capable.
3. Set selected offer ACCEPTED.
4. Set other competing offers to non-winning terminal states as needed.
5. Update request status to matched/closed behavior.
6. Create RideMatch with acceptedRideRequestOffer reference.
7. Notify both participants.

Result:
- one explicit commitment source
- no ambiguous multi-accept path

## 41. UML-to-Engineering Case Studies (Detailed Scenario Decomposition)

This section adds case-based decomposition to further strengthen the UML implementation narrative. Each case is structured as model statement -> engineering implementation -> verification evidence.

### 41.1 Case Study A: “A RideMatch Is Created Only After Explicit Acceptance”

Model statement:
- Activity and class-level notes indicate agreement is created only after acceptance branch.

Engineering implementation:
- Join request submission persists PENDING only.
- RideMatch creation is invoked in decision acceptance branch, not in submission path.
- Equivalent pattern applied for one-off offers: acceptance event creates match.

Verification evidence:
- Pending join request can exist without corresponding ride match.
- Match appears only after accept action from authorized actor.
- History and notifications update after acceptance transition.

Quality implication:
- This preserves transaction semantics and avoids false confirmation artifacts.

### 41.2 Case Study B: “Trust Before Acceptance”

Model statement:
- Supporting process requires trust checking before final acceptance actions.

Engineering implementation:
- Rider detail and one-off offer review surfaces include profile and rating summary.
- Trust data is displayed before request/accept action controls.

Verification evidence:
- UI order places trust panel above or adjacent to action buttons.
- DTO composition includes trust fields in decision contexts.

Quality implication:
- Decision quality is improved and model intent is respected in user experience.

### 41.3 Case Study C: “requestedSeats <= availableSeats”

Model statement:
- JoinRequest note enforces seat limitation against selected offer.

Engineering implementation:
- Frontend seat input constrained by available seats display context.
- Backend service revalidates requested seats at mutation time.
- Decision path also rechecks availability under current state.

Verification evidence:
- Invalid requests rejected with actionable messages.
- No match generated when constraint violated.

Quality implication:
- Constraint is enforced beyond UI assumptions.

### 41.4 Case Study D: “RideMatch Source XOR”

Model statement:
- RideMatch should reference one and only one accepted source family.

Engineering implementation:
- Match entity stores acceptedJoinRequest and acceptedRideRequestOffer links.
- Database check constraint enforces mutually exclusive source path.
- Unique constraints prevent duplicate source reuse.

Verification evidence:
- Admin match table shows one source populated and the other null.
- Invalid mixed-source persistence is blocked.

Quality implication:
- Agreement provenance is auditable and logically consistent.

### 41.5 Case Study E: “Driver Response Process In Scenario 2”

Model statement:
- Drivers view posted rider requests, submit offers, rider accepts one.

Engineering implementation:
- Driver hub displays open rider requests.
- Offer submission endpoint persists response records.
- Rider offer review route enables acceptance action.

Verification evidence:
- One-off request history shows offer counts.
- Accepted offer creates ride match and status progression.

Quality implication:
- Scenario 2 process is complete, not partial.

### 41.6 Case Study F: “Observability As Process Evidence”

Model statement:
- Process should be inspectable beyond hidden backend state.

Engineering implementation:
- My Trips segmented histories and filters.

Verification evidence:

Quality implication:
- Demonstration and marking can verify claims quickly.

### 41.7 Case Study G: “Concurrency-Sensitive Decision Paths”

Model statement:

Verification evidence:

Quality implication:
